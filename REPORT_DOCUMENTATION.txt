1. ABSTRACT
================================================================================

This project presents an Enhanced Indian Railway Train Delay Optimization System
that uses Constraint Programming (CP-SAT) to compute conflict-free, priority-
aware train schedules under real-world operational constraints including weather
disturbances, maintenance blocks, single-track bottlenecks, and crew regulations.
A post-hoc Explainable AI (XAI) layer using SHAP (SHapley Additive exPlanations)
with a gradient-boosted surrogate model identifies the primary drivers of delay
for each train in the optimized schedule, providing actionable insights for
railway operations managers.

Key results on a 13-station, 10-track, 8-train Indian railway network:
  - Optimal solutions found in under 1 second (0.06 s for Winter scenario)
  - 87.5% punctuality rate (7/8 trains on time) under Winter/No-maintenance
  - Weather delay identified as the dominant delay driver (SHAP importance: 6.30 min)
  - Priority-based overtaking correctly enforced across all 13 events in sample run

2. INTRODUCTION AND MOTIVATION
================================================================================

Indian Railways is one of the world's largest railway networks, carrying over
8 billion passengers annually. Train delays propagate through a tightly coupled
network, causing cascading effects that affect millions of commuters and billions
of rupees in freight. Existing timetabling tools used by Indian Railways (FOIS,
CRIS systems) are largely rule-based and do not model multi-factor interactions
(weather + maintenance + priority conflicts) simultaneously.

This system addresses the following research gap:
  - How can CP-SAT optimization model all major delay factors simultaneously?
  - How can the model enforce Indian Railway priority rules (Rajdhani > Shatabdi
    > Express > Mail > Passenger > Freight > Goods)?
  - How can SHAP explain post-hoc WHICH factors drove each train's delay?

The system is designed as a research and decision-support tool, not a real-time
deployment system.

8. STATION CONFIGURATION PARAMETERS

Each station has the following configuration fields used by the optimizer:

  Field               Type     Meaning
  ------------------  -------  ------------------------------------------------
  type                string   Station role (major_junction, crossing_station...)
  platforms           int      Number of main platforms (capacity)
  loops               int      Number of crossing/loop lines

  crew_change         bool     If True: adds 5 min dwell for superfast/express

  water_column        bool     If True + coaches > 16: adds 3 min dwell
  dwell_times         dict     Base minimum dwell per train type (minutes):
                               superfast, express, passenger, freight, mail_express, goods

DWELL TIME CALCULATION (_get_minimum_dwell_time()):
  base_dwell = station.dwell_times[train_type]
  if crew_change and type in [superfast, express, mail_express]: base_dwell += 5
  if maintenance_depot and type in [freight, goods]:             base_dwell += 2
  if coaches > 16 and water_column:                              base_dwell += 3
  return base_dwell  (used as lower bound for dwell integer variable)

PLATFORM CAPACITY:
  total_capacity = platforms + loops
  Used in AddCumulative constraint to prevent more trains than capacity at once.

STATION DWELL TIMES (sample, key stations):

9. DELAY MODELING PARAMETERS
================================================================================

DELAY TYPES (4 types tracked per train per station):
  weather       -- Weather-induced speed reductions on affected sections
  maintenance   -- Speed restrictions from track work (currently set to 0 in model)
  congestion    -- Queuing delays at shared sections (currently 0; capacity enforced
                   via interval constraints, not separate delay variable)
  operational   -- Peak-hour freight restrictions (up to 45 min per station)

DELAY VARIABLE BOUNDS:
  Each delay variable: IntVar(0, 180)  -- max 3 hours per type per station
  Planning horizon: 36 * 60 = 2160 minutes (36-hour window)

DELAY FACTOR (custom_delay_factor):
  Range: 0.0 to 2.0 (clamped)
  Default: 1.0
  Effect: Multiplies ALL weather additional_time values and maintenance speed_limits
  Use: Simulate increasingly severe conditions for sensitivity analysis

  delay_factor = 0.0  --> ideal conditions, no weather effect
  delay_factor = 0.5  --> moderate conditions
  delay_factor = 1.0  --> normal baseline (Indian Railways average)
  delay_factor = 1.5  --> high disruption
  delay_factor = 2.0  --> extreme conditions (cyclone/heavy blizzard)

GRADIENT FACTOR (in add_basic_operational_constraints()):
  Applied to base travel time for steep_climb sections:
    freight/goods:          gradient_factor = 1.4  (40% slower)
    superfast/express:      gradient_factor = 1.15 (15% slower)
    all others:             gradient_factor = 1.3  (30% slower)

FLEXIBILITY MULTIPLIER (max_travel_time upper bound):
  freight/goods:    flexibility_multiplier = 1.15
  all others:       flexibility_multiplier = 1.05
  if bottleneck:    flexibility_multiplier *= 1.10
  max_travel_time = int(min_travel_time * flexibility_multiplier) + 3

CROSSING BUFFER TIME (add_single_track_constraints()):
  is_bottleneck = True  -->  buffer_time = 3 minutes
  is_bottleneck = False -->  buffer_time = 2 minutes
  Purpose: Enforce safe headway between consecutive trains on single-track sections

MINIMUM SEPARATION AT STATIONS (crossing_rules):
  minimum_separation = 5 minutes
  Used in priority and overtaking constraints between departures

10. CONSTRAINT PROGRAMMING MODEL (OR-Tools CP-SAT)
================================================================================

SOLVER: Google OR-Tools CP-SAT (Constraint Programming - Boolean Satisfiability)
  - CP-SAT is a portfolio solver combining SAT, LP relaxation, and search
  - Solver parameters used:
      max_time_in_seconds  = 600  (10-minute hard timeout)
      num_search_workers   = 8    (parallel search threads)
      log_search_progress  = False
      linearization_level  = 2    (aggressive LP relaxation)

DECISION VARIABLES:
  For each train T and each station S in T's route:
    arr_{T}_{S}   : IntVar(0, 2160)  -- arrival time (minutes from midnight)
    dep_{T}_{S}   : IntVar(0, 2160)  -- departure time
    dwell_{T}_{S} : IntVar(0, 60)    -- dwell time at station

  For each train T, station S, and delay type D in {weather, maintenance,
                                                    congestion, operational}:
    {D}_delay_{T}_{S} : IntVar(0, 180)  -- delay contribution (minutes)

  For each train T, station S (if in station_config):
    platform_{T}_{S} : IntVar(1, platforms+loops)  -- assigned platform/loop

  Boolean decision variables:
    priority_order_{T1}_before_{T2}_{S} : BoolVar  -- ordering at station
    overtake_{T1}_{T2}_{S}              : BoolVar  -- overtaking decision
    train1_first_{T1}_{T2}_{S}          : BoolVar  -- priority enforcement
    high_departs_first_{Th}_{Tl}_{S}    : BoolVar  -- no-wait constraint
    freight_restriction_{T}_{S}_{P}     : BoolVar  -- peak-hour violation flag
    on_time_{T}                         : BoolVar  -- punctuality indicator

CONSTRAINT GROUPS:

  A. BASIC OPERATIONAL CONSTRAINTS (add_basic_operational_constraints):
     For each consecutive station pair (S_i, S_{i+1}) in route:
       arr[S_{i+1}] = dep[S_i] + actual_travel_time
       dep[S_i]    = arr[S_i] + dwell[S_i]    (for intermediate stations)
       dwell[S_i]  >= min_dwell(T, S_i)
       dep[first_station] = scheduled_departure + initial_delay  (initial_delay in 0..120)
       min_travel_time <= actual_travel_time <= max_travel_time

  B. WEATHER IMPACT CONSTRAINTS (add_weather_impact_constraints):
     For affected sections:
       weather_delay[T][S] = weather_additional * delay_factor
       (adjusted by train type: SUPERFAST=0, EXPRESS=20%, MAIL=50%, others=100%)
     For non-affected sections:
       weather_delay[T][S] = 0

  C. MAINTENANCE CONSTRAINTS (add_maintenance_constraints):
     Currently:  maintenance_delay[T][S] = 0  for all T, S
     (Maintenance modeled through travel time bounds, not separate delay vars)

  D. SINGLE-TRACK CONSTRAINTS (add_single_track_constraints):
     For each single-track section (from, to):
       Create interval var for each train using that section
       Apply AddNoOverlap(all intervals on that section)
     This prevents simultaneous use of single-track sections.
     Buffer time (2-3 min) is added to each interval end.

  E. PRIORITY CONSTRAINTS (add_priority_constraints):
     For each pair (T_high, T_low) with priority_diff >= 0.5 at shared station:
       EITHER: dep[T_high][S] + min_sep <= dep[T_low][S]
       OR:     dep[T_low][S] + 90 <= arr[T_high][S]   (large gap if reversed)
     Overtaking:
       IF overtake: dep[T1][S] + min_sep <= dep[T2][S]
       IF NOT:      dep[T2][S] + min_sep <= dep[T1][S]

  F. NO-WAITING-FOR-LOWER-PRIORITY (add_no_waiting_for_lower_priority_constraints):
     High-priority train either departs before low-priority OR
     departs at least 120 minutes AFTER (large separation to avoid deadlock).

  G. PLATFORM CAPACITY CONSTRAINTS (add_platform_capacity_constraints):
     For each station with more trains than platform+loop capacity:
       AddCumulative(interval per train, demand=1, capacity=platforms+loops)

  H. FREIGHT RESTRICTIONS (add_freight_restrictions):
     For each freight/goods train at each station during peak hours:
       IF in peak period: operational_delay >= 45 min

11. OBJECTIVE FUNCTION DESIGN
================================================================================

The objective is a weighted minimization of multiple penalty terms:

  MINIMIZE:
    delay_penalty        +   (primary: penalizes all delays, weighted by priority^2)
    bottleneck_penalty   +   (3x multiplier at bottleneck stations)
    premium_protection   +   (SUPERFAST: 50000x, EXPRESS: 10000x per delay minute)
    dwell_penalty        +   (extra dwell above minimum, weighted by priority^2)
    operational_efficiency - (gap between consecutive trains at same station)
    punctuality_bonus        (bonus for being on time, SUPERFAST: 100000, EXPRESS: 30000)

WEIGHTS BY TRAIN TYPE:
  SUPERFAST:    weight = 1000
  EXPRESS:      weight = 200
  MAIL_EXPRESS: weight = 80
  PASSENGER:    weight = 15
  FREIGHT/GOODS:weight = 3

DELAY PENALTY:
  delay_penalty += total_delay_at_station * int(priority^2 * weight * 100)

BOTTLENECK PENALTY:
  if station is a bottleneck exit:
    bottleneck_penalty += total_delay * int(priority^2 * weight * 3.0 * 150)

PUNCTUALITY:
  On-time = arrival at final station <= scheduled_departure + 120 + 5 minutes
  Bonus: SUPERFAST=100000, EXPRESS=30000, MAIL=10000, others=priority*1000

RATIONALE:
  The quadratic priority weighting (priority^2) ensures Rajdhani (5.0^2 = 25x)
  is protected far more aggressively than Coal Goods (1.0^2 = 1x), reflecting
  Indian Railways' actual priority hierarchy.

14. EXPLAINABLE AI MODULE (SHAP)
================================================================================

14.1 MOTIVATION
  The CP-SAT optimizer produces optimal schedules but is a black-box
  from the perspective of delay attribution. A station manager cannot directly
  read the CP model to understand "WHY is Train X delayed by 15 minutes?"
  SHAP addresses this by explaining predictions of a surrogate model that
  approximates the optimizer's behavior.

14.2 APPROACH: SURROGATE MODEL + SHAP TREEXPLAINER
  Step 1 -- Data generation:
    Run the optimizer 10 times at delay factors [0.2, 0.4, ..., 2.0]
    with same scenario_date and maintenance_preset.
    Each run produces 8 train observations (8 trains x 10 runs = 80 rows).
    Total training samples: ~80 (varies if any run fails to find a solution).

  Step 2 -- Feature extraction (_extract_train_features):
    For each train-solution pair, extract a 9-dimensional feature vector:
      [0] weather_delay      : sum of weather delays across all route stations
      [1] maintenance_delay  : sum of maintenance delays across all route stations
      [2] congestion_delay   : sum of congestion delays across all route stations
      [3] operational_delay  : sum of operational delays across all route stations
      [4] bottleneck_sections: count of bottleneck tracks in the train's route
      [5] steep_grade_sections: count of steep_climb tracks in the train's route
      [6] single_track_sections: count of single-track segments in the train's route
      [7] train_priority     : train priority score (float, 1.0 to 5.0)
      [8] delay_factor       : the global delay multiplier used in that run

    Target (y): train_total_delay (integer, minutes)

  Step 3 -- Surrogate model:
    Algorithm: sklearn.ensemble.GradientBoostingRegressor
    Hyperparameters:
      n_estimators  = 150
      max_depth     = 3
      learning_rate = 0.1
      random_state  = 42

  Step 4 -- SHAP computation:
    explainer = shap.TreeExplainer(model, X_train)
    shap_values = explainer.shap_values(X_current)  -- shape: (n_trains, 9)

    TreeSHAP algorithm (Lundberg et al., 2020) computes exact Shapley values
    for tree ensembles in polynomial time O(TLD^2) where T=trees, L=leaves,
    D=depth. This is the preferred method for gradient-boosted trees.

  Step 5 -- Global importance:
    global_importance[feature] = mean(|shap_values[:, feature_idx]|)
    Ranks features by average magnitude of their contribution across all trains.

  Step 6 -- Per-train SHAP breakdown:
    For each train:
      baseline (expected_value) + sum(shap_values) ≈ predicted_delay
    Each SHAP value is the marginal contribution of that feature above baseline.
    Positive SHAP = increases delay. Negative SHAP = reduces delay.

14.3 FEATURE ENGINEERING RATIONALE
  - weather_delay, maintenance_delay etc. are summed across stations (not per-
    station) to give a route-level signal. This is intentional: SHAP works best
    with a small number of well-defined features.
  - bottleneck_sections is route-structural: it captures HOW MANY bottleneck
    segments a train must traverse, regardless of scenario.
  - delay_factor is included to allow the surrogate to learn global severity
    sensitivity — critical for generalizing across the 10 training runs.

14.4 LIMITATIONS OF THE XAI APPROACH
  - The surrogate model R^2 depends on how well GBR approximates the CP solution
    space. For this 8-feature, 80-sample problem, fit is typically near-perfect.
  - SHAP explains the SURROGATE MODEL, not the CP solver directly. This is
    a standard XAI trade-off (model-agnostic explanation via surrogate).
  - With only 10 delay_factor variations, feature interactions involving
    maintenance may be undersampled if maintenance runs all produce the same
    delay structure (all zeros in current model).

17. ALGORITHM COMPLEXITY AND PERFORMANCE
================================================================================

CP-SAT COMPLEXITY:
  CP-SAT is NP-hard in the general case (job-shop scheduling variant).
  However, the problem is small: 8 trains, 13 stations, ~200 variables.
  Empirical solve times:
    - Normal scenario (no maintenance):   0.06 seconds
    - Heavy maintenance (preset 5):       < 1 second
    - Emergency repair (preset 6):        < 2 seconds
  The solver finds OPTIMAL solutions in all tested configurations.

SHAP SURROGATE TRAINING:
  Training data generation: 10 optimizer runs x ~1 second each = ~10 seconds
  GradientBoosting training: <1 second (80 samples, 150 estimators)
  TreeSHAP computation: <0.1 seconds
  Total SHAP overhead: ~12-15 seconds

TOTAL RUNTIME (Full Detailed Report, Mode 4):
  ~15-20 seconds including SHAP surrogate generation

SCALABILITY NOTES:
  - The 8-train, 13-station model solves trivially.
  - For 50+ trains or 100+ stations, CP-SAT may need:
      - Longer time limit (currently 600 s)
      - Decomposition (e.g., rolling horizon)
      - Fewer binary overtaking variables
  - SHAP scaling: TreeSHAP is O(TLD^2) -- fast for GBR depth=3

18. KEY OBSERVATIONS AND FINDINGS
================================================================================

1. PRIORITY PROTECTION WORKS CORRECTLY:
   Rajdhani (priority 5.0) is always on time across all tested scenarios.
   CP-SAT correctly enforces that lower-priority trains yield to higher-priority
   trains at all shared stations and single-track sections.

2. SINGLE-TRACK BOTTLENECK IS THE DOMINANT STRUCTURAL CONSTRAINT:
   The Bhor Ghat section (Station_A to Junction_Z via Lonavala) creates a
   sequential bottleneck. Even with 0 weather delays, trains must queue through
   this section in strict priority order. This mirrors real-world Bhor Ghat
   operations on Indian Western Railway.

3. WEATHER AFFECTS LOWER-PRIORITY TRAINS DISPROPORTIONATELY:
   The train-type weather reduction policy (Superfast=0%, Express=20%,
   others=100%) means that in monsoon/winter scenarios, passenger and freight
   trains absorb the full weather delay while premium trains are insulated.
   SHAP confirms this: Weather SHAP for passenger = +9.14 min,
   vs. Superfast = -5.86 min (SHAP negative = below baseline = protected).

4. FREIGHT TRAINS ARE STRUCTURALLY DISADVANTAGED:
   Goods/freight trains have: lowest priority (1.0-1.5), full weather delays,
   peak-hour restrictions (07:00-10:00), and highest dwell times at stations.
   This matches Indian Railways policy of prioritizing passenger traffic.

5. SHAP BASELINE INTERPRETATION:
   The 6.2-minute baseline (expected_value) represents the average delay across
   all 80 training observations (8 trains x 10 delay factors). A SHAP value
   of -5.86 for a Superfast train means: "This train is predicted to be 5.86
   minutes LESS delayed than the average train in the dataset, due to its
   weather protection."

6. MAINTENANCE HAS ZERO SHAP IN CURRENT MODEL:
   Because maintenance_delay is hardcoded to 0 in the CP model for all runs,
   the surrogate never observes maintenance variation, and SHAP correctly
   assigns zero importance to it. To properly analyze maintenance impact,
   the model must be extended to vary maintenance_preset across training runs.

19. LIMITATIONS AND FUTURE WORK
================================================================================

CURRENT LIMITATIONS:
  1. Maintenance delay variables are all set to 0 in add_maintenance_constraints().
     Maintenance only affects travel time bounds indirectly. Future: implement
     speed-limit-to-delay conversion for blocked sections.

  2. The network is a single-corridor abstraction. Real Indian Railways has
     thousands of stations and hundreds of intersecting routes.

  3. Crew constraints (max 8-hour duty, 30-minute rest) are defined in
     operational_constraints but NOT added to the CP model as formal constraints.
     They appear only in reporting text.

  4. SHAP training varies only delay_factor across 10 runs. A more robust
     approach would also vary maintenance_preset and weather_scenario to
     produce richer training data for the surrogate model.

  5. The system is offline/batch -- not designed for real-time rescheduling.

FUTURE WORK:
  1. Implement real maintenance delay constraints in the CP model.
  2. Extend to rolling-stock availability and locomotive assignment.
  3. Add real-time delay injection (live sensor feeds).
  4. Use LSTM/Transformer for delay prediction as an alternative surrogate.
  5. Benchmark against CRIS (Indian Railways' scheduling system) data.
  6. Multi-commodity flow integration for freight optimization.
  7. Extend SHAP training to include maintenance_preset and season as variables.
  8. Deploy as a web API for operations control center use.
